# `audit/impeccable/` — design anti-pattern ratchet

Decremental count baseline for [`impeccable`](https://www.npmjs.com/package/impeccable) (devDep in `@fafa/frontend`, see PR #597) wired as a CI block-new gate.

## Files

| Path | Role |
|------|------|
| `baseline.json` | Committed SoT of the **maximum** allowed counts per category and per file. Total + per-category + per-file. Regenerated by maintainers when a category is brought down. |
| `baseline.schema.json` | JSON Schema for `baseline.json` shape (informational — not validated in CI today). |

## How it works

1. CI workflow `.github/workflows/impeccable-ratchet.yml` runs `impeccable design:detect:json --fast` on every PR touching `frontend/**` or design tokens.
2. `scripts/audit/impeccable-ratchet-check.mjs` compares the current report against `baseline.json` and **fails the PR** if:
   - total count grew, OR
   - any category count grew, OR
   - a file that had 0 issues in baseline now has ≥1 issue.
3. Fixing PRs run `scripts/audit/impeccable-baseline-write.mjs` to ratchet the baseline DOWN. The script refuses to write a higher count (use `--allow-increase` only for the initial bootstrap, never afterwards).

This is the same pattern as the [Contract Drift Ratchet](../../.github/workflows/contract-drift-ratchet.yml) (PR-7a) and the [Registry new-file gate](../../.github/workflows/registry-new-file-gate.yml) (ADR-058 PR-G).

## Regenerate locally

```bash
# 1. capture current state
npm --workspace=@fafa/frontend run design:detect:json > /tmp/cur.json

# 2. check (no write)
node scripts/audit/impeccable-ratchet-check.mjs --json-file /tmp/cur.json

# 3. ratchet down (only if all dimensions ≤ existing baseline)
node scripts/audit/impeccable-baseline-write.mjs --from /tmp/cur.json
```

## Local pre-commit hook (Phase 1: warn-only)

A Husky pre-commit hook (`.husky/pre-commit`) runs the ratchet locally when a commit touches `frontend/**`, `packages/design-tokens/**`, `audit/impeccable/**`, or any of the ratchet's own files. It is **warn-only** in Phase 1 — it prints a notice if your change increases the count but does NOT block the commit. The CI workflow `.github/workflows/impeccable-ratchet.yml` is the authoritative BLOCKING gate.

Hook env vars:
- `IMPECCABLE_HOOK=skip` — silence the hook entirely (e.g., commits unrelated to design)
- `IMPECCABLE_HOOK=block` — promote to blocking (planned default in Phase 2 once the cascade closes)
- `IMPECCABLE_HOOK=warn` — default (print warning on regression)

The hook fires impeccable in `--fast` mode (skip Chromium download via `.npmrc`), so cost is ≤ 5s on average for affected commits.

## Anti-bricolage discipline

- **Never** raise the baseline to dodge a CI failure (use `--allow-increase` only at initial bootstrap).
- **Never** add files to a per-file ignore list — fix the actual anti-pattern.
- **Never** wrap impeccable in a project-specific shim — the canonical CLI is the authority.

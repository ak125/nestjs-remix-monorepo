# `audit/` — deep-audit map of the monorepo (Phase 0.9)

> **PR-0a generated this. It deletes nothing and changes no behaviour — it is a
> verifiable map of the runtime, the dette, and the risks.** The map drives the
> later phases (ratchet, cleanup-by-domain, cycles/boundaries, CI blocking).

Regenerate everything with:

```bash
npm run audit:inventory          # rebuilds audit/* (+ the gitignored cache)
npm run audit:inventory:check    # rebuild + `git diff --exit-code -- audit/*.json` (CI freshness gate)
```

The generator is [`scripts/audit/build-deep-inventory.js`](../scripts/audit/build-deep-inventory.js)
(plain Node — no `tsx`/`ts-morph` runtime dep). It is **deterministic**: no
timestamps in the output, every array sorted by a stable key, so re-running on
the same checkout produces byte-identical files.

## Data sources (no naive grep for structural analysis)

| Source | Used for |
|---|---|
| `dependency-cruiser --output-type json` | AST import graph (static + dynamic), orphans, `no-deep-module-access` violations |
| `madge` (programmatic) | circular dependencies |
| `knip --reporter json` | unused files / exports / types / duplicate exports |
| TypeScript compiler API (`typescript`) | NestJS decorators (class ↔ file), `@Module({...})` arrays (DI reachability), Remix `loader`/`action` exports |
| `git ls-files` / `git log` | file enumeration, deterministic `last_modified` (last commit date, **not** fs mtime) |
| static tables in the generator | domain routing, never-auto-delete zone, devops surface |

## Conditions of deletion (#0–#8) — no file is removed without all of them

0. **Not in the never-auto-delete zone** (`runtime-entrypoints.json#never_auto_delete_globs`) — hard refusal, whatever the audit says.
1. Zero static import (`dependency-cruiser` + `knip` + grep).
2. Zero dynamic import (`dynamic-import-edges.json`).
3. Zero runtime use (`runtime-entrypoints.json`).
4. Zero string reference (grep of path / basename / registry-by-string).
5. Zero CI/script use (`.github/workflows/**`, `package.json` `scripts`/`bin`, `.husky/**`, `docker/**`).
6. Zero DB/cron/worker use (Supabase migration / cron / BullMQ processor; for data services, the RPC/table is itself a candidate orphan — real DB drops go through the RPC Gate + a vault ADR, never here).
7. Build + tests OK after removal (`npm run build` ×2, `npm test`).
8. Human review (the PR).

**Responsibility split (no god-script):** `audit/*.json` = source of truth for the analysis (conditions 0–6, computed here). [`scripts/cleanup/validate-before-delete.sh`](../scripts/cleanup/validate-before-delete.sh) = the *last local guard-rail before `git rm`* — it re-checks the simple conditions 0–6 mechanically on the targeted file (it consults `runtime-entrypoints.json` and `dynamic-import-edges.json` when present); it is **not** a re-run of the audit. Conditions 7–8 happen at PR time.

## Relationship to `audit-reports/`

`audit-reports/` (the **ratchet** — `phase0-baseline.json` + `audit-compare-baseline.js` + the raw tool snapshots) and `audit/` (this **deep-audit map** — the decision artefacts) coexist intentionally. The ratchet stops *new* dette; the map plans *paying down* the existing dette.

## How the map drives the next phases

- **PR-1 — ratchet:** `dead-code-candidates.json` is the work-tracking artefact; thresholds tighten as it shrinks.
- **PR-2 — cleanup hors runtime:** the `confidence: high`, non-runtime, non-`backend/src/modules/**`, non-`frontend/app/routes/**` candidates.
- **PR-3 — backend NestJS-aware:** make `knip.json` see NestJS DI, then triage the real backend dead code (`config/`, `common/`, the `nestjs_unreachable_modules` subtrees).
- **PR-4 — frontend Remix-aware:** the `frontend-shared` candidate cluster.
- **PR-5 — cycles:** `cycle-map.json` in priority order.
- **PR-6 — boundaries:** `module-boundaries.json#deep_access_violations`; `runtime-entrypoints.json` feeds the `allowedOrphans` allowlist.
- **PR-0b** added `db-usage-map.json` (`.from()`/`.rpc()` ↔ migrations, *candidate* orphan tables/RPC only — no verdicts), `risk-register.md`, `cleanup-plan-by-domain.md`. `cleanup-plan-by-domain.md` is the authoritative driver for PR-2 → PR-6; the DB/RPC surface goes to a dedicated vault ADR + the RPC Gate, never a code PR.
